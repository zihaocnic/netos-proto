# 工作流（固定流程）

> 目的：确保“编码可追溯、可审查、可恢复”，并避免断档。

## 角色分工
- **Codex**：负责具体代码编写，并自行提交（commit）。
- **我（助手）**：负责审查、验证、与用户沟通；必要的文档/流程调整由我提交。

## 固定流程（每个任务都按此执行）

### 0) 启动阶段
- 与你确认任务范围、完成标准。
- 明确是否需要 Codex 执行编码。

### 1) Codex 编码阶段
- Codex 编写代码。
- **谁改谁 commit**：Codex 修改的代码由 Codex 自己提交。

### 2) 审查与验证
- 我对 Codex 的提交进行审查（逻辑与风险）。
- 运行必要验证/脚本，输出关键日志。

### 3) 先汇报 → 再请求 Push
- 我先给出**任务结果汇报**（含验证结论）。
- **只有你确认**，我才会 push。

### 4) Push 后处理
- push 完成后：
  - 更新 `docs/CONTINUITY.md` / `docs/STATUS.md`
  - 汇报**全局推进情况 + 下一步计划**

## 永久规则
- **谁改谁 commit**（Codex 写的由 Codex commit；我改文档由我 commit）。
- **未确认不得 push**。
- **每次任务完成后必须更新断档恢复文档**。
