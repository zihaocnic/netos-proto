# NetOS Agent Context

本文件用于给后续轮次的 Codex / agent 提供稳定上下文，避免因会话重启丢失项目背景。

## 项目目标

基于 `/home/ubuntu/Project/NetOS` 下的会议纪要、设计文档与路线图，持续迭代出一个 **可运行的 NetOS 系统原型**。

当前阶段目标不是做深，而是：

1. 先做出 **可运行 demo**
2. 先保证 **容器化可启动**
3. 先形成 **清晰模块边界**
4. 再逐步贴近会议纪要中的目标架构

## 当前已确定原则

- 产出目录固定为：`/home/ubuntu/Project/NetOS/netos-proto`
- 仓库已初始化并同步到 GitHub
- 开发过程中应持续使用 git 管理阶段成果
- 如无必要，不频繁打扰用户确认
- 重要架构分歧、方向性变化、或阻塞问题再提问

## 代码与目录约束

- 不要把所有源码平铺在仓库根目录
- 源码必须按模块拆分，保持可读性
- 文档、设计、基础设施、源码分目录维护
- 当前推荐目录：
  - `docs/`：项目说明、技术选型、阶段计划、agent briefing
  - `design/`：架构与设计说明
  - `infra/`：Docker / Compose / 启动脚本 / 拓扑配置
  - `src/`：源码

## 当前技术方向

依据会议纪要与设计文档，当前阶段采用：

- **语言**：C++20
- **缓存后端**：Redis（代理模式）
- **部署方式**：Redis + Sync App 同容器
- **原型形态**：Docker Compose 可运行 demo
- **当前网络实现**：先用轻量级 UDP 请求/数据同步验证基本链路

## 当前范围控制

当前只做 Phase 1 / demo-first：

- 先完成 Pull 路径的最小闭环
- 先验证多节点间请求、响应、落本地缓存
- 暂不急于一次性做全量 Bloom Filter、复杂 Push 机制、完整拓扑自适应
- 暂不追求论文级性能，只追求结构正确、可以演示、可继续扩展

## 后续开发优先级

后续轮次优先考虑：

1. 强化模块边界与代码可维护性
2. 让 QueryTable / SyncTable / Node pipeline 更贴近会议纪要中的语义
3. 增加可观测性（日志、验证脚本、简单测试）
4. 逐步引入更接近 NetOS 的状态机结构
5. 再考虑 Bloom Filter stub / topology config / 更复杂拓扑演示

## Agent 工作方式

每一轮 agent / Codex 启动时：

1. 先阅读本文件
2. 再阅读：
   - `docs/TECH_STACK.md`
   - `design/ARCHITECTURE.md`
   - `docs/ROADMAP.md`
   - 最新 `README.md`
3. 查看最近 git log
4. 以“增量、可运行、可提交”为原则推进

## Git 纪律

- 每完成一个明确阶段，就提交一次 git commit
- commit message 应反映阶段意图，而不只是文件变化
- 任何让 demo 无法运行的改动都不应直接结束任务

## 面向用户的交付原则

- 永远优先交付“能跑的东西”
- 解释要清楚，但不要空谈
- 如果必须二选一，优先选择：
  - 更容易运行
  - 更容易看懂
  - 更容易继续迭代
